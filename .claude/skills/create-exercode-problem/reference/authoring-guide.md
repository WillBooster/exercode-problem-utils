# 良い問題を作るためのノウハウ

WillBooster の教材生成（`gen-em` の作問ガイドライン）と実運用から抽出した、教育的に良質で
機械採点に耐える問題を作るための原則。**設計段階で必ず目を通すこと。**

## 鉄則: 機械採点できない問題を作らない

Exercode は事前に用意したテストケース（入力と期待出力のペア）で自動採点する。したがって:

- **入力に対して出力が一意に定まる**問題、または**プログラムで正誤を判別できる**問題だけを作る。
- 人によって答えが変わる問題は禁止（例: 「あなたの名前を出力せよ」は人によって出力が違うため採点不能）。
- 実行途中でユーザの追加入力や操作を要求する**インタラクティブな問題は採点不能**なので禁止。
- **機械採点できない問題の作成は禁止。** 設計が採点可能か必ず先に確認する。

判定が一意にならない場合の回避策:

- 出力形式を厳密に指定して一意化する（桁数・区切り・順序・改行を明示）。
- 一意でない部分は `command`/自作 judge で正規表現・数値抽出・集合比較などの**緩い判定**にする。
- LLM 問題（`llm` プリセット）では「完全一致」でなく「部分一致・含意・長さ」で判定する。

## 難易度設計（初級 / 中級 / 上級 / 超級）

新しい概念を説明するたびに問題を出す。講義の最後にまとめて出さない。1セクションに複数難易度を混ぜてよい。

- **初級**: 学んだ概念を使って基本的なプログラムが書ける。**数を十分に用意し**、基礎の定着を最優先。
- **中級**: 学んだ概念を応用し、新しい状況・課題に対応できる。
- **上級**: 複数の学習内容を組み合わせ、より複雑なプログラムを作れる。
- **超級**: 未知の課題・実践的問題に、学んだ知識を活用して自力で解決策を考えられる。挑戦的でよい（中級以上は解けなくても学習進行に支障がない応用課題とする）。

学習者が**すでに学んだ範囲だけ**で解けるように設計する。まだ教えていない構文・関数を前提にしない。

## 問題文の書き方

- 簡潔・明確な日本語で、初学者にも理解できるように書く。
- 「プログラミングの第一歩として」のような**問題解決に不要な前置き・装飾は書かない**。
- 専門用語は使う前に定義する。
- **問題文の本文に入出力例を混ぜない。** 例は `## 入力例N` / `## 出力例N` セクションに分けて書く（後から例と解説を追記する運用のため）。
- 入力形式・出力形式を曖昧さなく規定する（区切り文字、改行、桁、順序、大文字小文字など）。
- 標準的な見出し構成に従う: `## 問題文` / `## 制約` / `## 入力` / `## 出力` / `## 入力例N` / `## 出力例N`。

## 「特定構文の習得」が目的のときの締め方

ある構文・関数を使わせるのが目的の問題では、抜け道を塞ぐ:

- `## 制約` に明記する。例:「`for`文を使ってください」「`**`演算子を使ってください」「`abs`関数を使ってください」。
- 抜け道の使用を禁止する。例:「`while`文を使ってはいけません」「`eval`関数を使ってはいけません」「`sum`関数を使ってはいけません」。
- frontmatter で機械的にも強制する:
  - 必須構文 → `requiredRegExpsInCode`（例: `['\bfor\b']`）
  - 禁止構文 → `forbiddenRegExpsInCode`（例: `['\bsum\s*\(']`）/ `forbiddenTextsInCode`
- これにより、目的の構文を使わない解答は REQUIRED/FORBIDDEN_PATTERNS_IN_CODE_ERROR で弾ける。

注: 正規表現はコメントや文字列リテラルを誤検知しないよう注意（本ライブラリには `removeCommentsInSourceCode` ヘルパがある）。緩すぎず厳しすぎない表現にする。

## テストケース設計

- **最低4個以上**作る。テストケースが少ないと誤って正解と判定するリスクが高まる。
  - コンテスト形式では `example_` 2個（表示）＋ `test_` 2個（非表示）が目安。
- 観点を分けて網羅する:
  - `small` … 仕様どおりの基本ケース。
  - `large` … 大きい値（オーバーフローや効率の確認。例: 制約上限 `10^9`）。
  - `edge` … 境界・特殊値（`0`、上限、最小、片方だけ大きい、同値、符号、空配列など）。
- model answer の**誤った変種が必ず落ちる**ようにテストケースを設計する。`% 10` だけ間違えた解答が通ってしまうなら、その差が出る入力を足す。
- 期待出力は、採用する採点方式の比較セマンティクスを踏まえて作る（例: 標準入出力ベースの採点は空白区切りトークン比較・浮動小数点は誤差 1e-6 許容。自作 judge なら比較規則は自分で決める）。
- 計算量の制約（time/memory limit）が解法を適切に選別するか確認する。意図した計算量でないと通らないなら、それを誘導する大きな入力を入れる。

## model answer と失敗解答

- **正解を最低1つ**用意し、全テストケースで受理されることを必ず確認する（model answer が落ちる＝問題かテストケースのバグ）。
- 複数言語で出すなら各言語の `model_answers/<lang>/` を用意。
- 余裕があれば **棄却されるべき解答**（`model_answers.fails/` や `model_answers.test/<lang>_<code>/`）を用意し、judge が正しく不合格にすることを確認する。これがテストケースの十分性の検証になる。

## 実行環境の選択（コスト順）

- **標準入出力（コマンドライン）**: 任意の言語が使え、最も低コスト・高速。**可能な限りこれを選ぶ**。Node.js 互換で書ける問題をわざわざ Web/GUI にしない。
- **Web（ブラウザ）**: テストが重い。HTML/CSS/JS の挙動を判定したいときだけ使う。判定は自作 judge で行う（実例 `example/web_page_weather`: `startHttpServer` で cwd を配信し puppeteer で DOM 検査）。
- **GUI（デスクトップ）**: スクリーンショット判定でさらに重い。`guiCommand` プリセットを使い、時間制限を長め（例 5000ms）にする（実例 `example/gui_python_window`）。

具体的な機構・API は `judge-presets.md` を参照。

## モジュール/インポート問題での補助ファイル

学習者に特定のファイルを渡したい場合は、**テンプレートコード（初期コード）として提供**する
（judge サーバのコンテスト形式では `templates/<言語ID>/`）。問題文に、テンプレートに含めるべきファイル情報を明記する。

## 最終チェック（作問者の自己点検）

- 入力→出力が一意、またはプログラムで正誤が決まる（機械採点可能）。
- 問題文に曖昧さがなく、入出力形式が厳密。前置き・例の混入がない。
- 難易度と学習目標が一致し、未習の知識を前提にしていない。
- テストケース4個以上、small/large/edge を網羅、誤った解答が落ちる。
- model answer が1つ以上あり全ケースで受理される。
- 特定構文が目的なら制約＋正規表現で抜け道を塞いでいる。
- time/memory limit が解法に対して妥当。
